Coveo component Analytics

This component logs all user actions performed in the search interface and sends them to a REST web service exposed through the Coveo Cloud platform.
You can use data to evaluate how users are interacting with the search interface, improve relevance and produce analytics dashboards in the Coveo platform.

Send Custom Events

In some scenarios, you want to send custom data to the Coveo Cloud analytics (see Coveo Cloud Usage Analytics). The Coveo JavaScript Search Framework offers helpers to communicate with the Coveo Analytics REST API, so you do not have to write code to call the API directly.

  1. First, you need to craft your custom event cause and meta.
    NB: The event names must be unique.

    // customEventType allows to regroup similar event types together when doing reporting (e.g., search box).
var customEventCause = {name: 'customEventName', type:'customEventType'};
var metadata = {key1: "value1", key2:"value2"};

  2. Send your custom event.

    Coveo.logCustomEvent(document.querySelector('#search'), customEventCause, metadata);
  // OR (using the jquery extension)
$('#search').coveo('logCustomEvent', customEventCause, metadata);

  3. Send your custom search event
    NB: If you want to log a searchEvent, be sure to always call the helper before you call executeQuery.

    function myCustomButtonWasClicked() {
  Coveo.logSearchEvent(document.querySelector('#search'), customEventCause, metadata);
  Coveo.executeQuery(document.querySelector('#search'));
  // OR (using the jquery extension)
  $('#search').coveo('logSearchEvent', customEventCause, metadata);
  $('#search').coveo('executeQuery');
}

  4. Send a custom searchAsYouType event
    NB: If you want to log a searchAsYouTypeEvent, be sure to always call the helper before you call executeQuery.

    function myCustomButtonWasClicked() {
  Coveo.logSearchAsYouTypeEvent(document.querySelector('#search'), customEventCause, metadata);
  Coveo.executeQuery(document.querySelector('#search'));
  // OR (using the jquery extension)
  $('#search').coveo('logSearchAsYouTypeEvent', customEventCause, metadata);
  $('#search').coveo('executeQuery');
}

  5. Send a custom click event.

    Coveo.logClickEvent(document.querySelector('#search'), customEventCause, metadata, result);
// OR (using the jQuery extension)
$('#search').coveo('logClickEvent', customEventCause, metadata, result);

Component Options

anonymous

anonymous: boolean
  • Defined in src/ui/Analytics/Analytics.ts :119

Specifies whether the search user identities are converted in a unique hash in the logged analytics data to prevent analytics reviewers and managers to identify who performs which queries.
When enabled, the Coveo Analytics Platform can still properly identify sessions made by anonymous users, versus ones from users that are authenticated in some way with the site containing the search page.
The default value is false.

Markup configuration example(s) :
data-anonymous='true'
data-anonymous='false'

endpoint

endpoint: string
  • Defined in src/ui/Analytics/Analytics.ts :113

Specifies the URL of the analytics logger for rare cases where it is different from the default usage analytics Coveo Cloud endpoint (https://usageanalytics.coveo.com).

Markup configuration example(s) :
data-endpoint='foo'

organization

organization: string
  • Defined in src/ui/Analytics/Analytics.ts :143

Specifies the organization bound to the access token. This is necessary when using an access token because it can be associated with more than organization. If this parameter is not specified, it will fallback on the organization used for the search endpoint.

Markup configuration example(s) :
data-organization='foo'

searchHub

searchHub: string
  • Defined in src/ui/Analytics/Analytics.ts :125

Sets the Search Hub dimension on the search events.
The Search Hub dimension is typically a name that refers to a specific search page. For example, one could use the CommunitySite value to refer to a search page on a company's public community site.
The default value is default.

Markup configuration example(s) :
data-search-hub='foo'

splitTestRunName

splitTestRunName: string
  • Defined in src/ui/Analytics/Analytics.ts :131

Specifies the name of the split test run that the search page is a part of.
This dimension can be used to perform A/B testing using different search page layouts and features, inside the Coveo Query pipeline.
By default, this value is not specified and no split test run name is reported to the Coveo Analytics Platform.

Markup configuration example(s) :
data-split-test-run-name='foo'

splitTestRunVersion

splitTestRunVersion: string
  • Defined in src/ui/Analytics/Analytics.ts :137

Specifies the version name for the page when a split test run is active.
When reporting on A/B testing analytics data, this value specifies the test run version name that has been presented to the user.
By default, this value is not specified.

Markup configuration example(s) :
data-split-test-run-version='foo'

token

token: string
  • Defined in src/ui/Analytics/Analytics.ts :109

Specifies the token used to gain access the analytics endpoint.
This attribute is optional, the component will use the search token by default.

Markup configuration example(s) :
data-token='foo'

logSearchAsYouType

  • logSearchAsYouType<T>(actionCause: IAnalyticsActionCause, meta: T): void
    • Defined in src/ui/Analytics/Analytics.ts :206

    Log a search as you type event on the service, using a cause and a meta object.
    This is extremely similar to a search event, except that search as you type, by definition, will be frequently called.
    The PendingSearchAsYouTypeEvent will take care of logging only the "relevant" last event: After 5 seconds of no event logged, or after another search event is triggered somewhere else in the interface.
    This is to ensure that we do not needlessly log every single partial query, which would make the reporting very confusing.

    Type parameters

    • T

    Parameters

    • actionCause: IAnalyticsActionCause
    • meta: T

      Can be an empty object ( {} ).

    Returns void

logSearchEvent

  • logSearchEvent<T>(actionCause: IAnalyticsActionCause, meta: T): void
    • Defined in src/ui/Analytics/Analytics.ts :194

    Log a search event on the service, using a cause and a meta object.
    Note that the event will be sent on the service when a query successfully return, not immediately after calling this method.
    Normally, this should be called using the following "format":
    usageAnalytics.logSearchEvent({name : 'foo', type : 'bar'}, {'key':'value'});
    this.queryController.executeQuery();
    This will queue up an analytics search event. Then the query is executed. The search event will be sent to the service when the query is successfully completed.

    Type parameters

    • T

    Parameters

    • actionCause: IAnalyticsActionCause
    • meta: T

      Can be an empty object ( {} ).

    Returns void
