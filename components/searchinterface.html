Coveo component SearchInterface

This component is the root and main component of your search interface.
You should place every other component inside this component.
It is also on this component that you call the initialization function.
Since this component is the root of your Search UI, it is recommended that you give it a unique HTML id attribute in order to reference it easily.

autoTriggerQuery

autoTriggerQuery: boolean
  • Defined in src/ui/SearchInterface/SearchInterface.ts :132

Specifies whether the init function should trigger the first query automatically when the page is loaded.
Note that if you set this to false, then the hideUntilFirstQuery option still applies. This means that the animation will still show until a query is triggered.
Default is true.

Markup configuration example(s) :
data-auto-trigger-query='true'
data-auto-trigger-query='false'

enableDuplicateFiltering

enableDuplicateFiltering: boolean
  • Defined in src/ui/SearchInterface/SearchInterface.ts :158

Specifies whether to filter duplicates on the search results.
When true, duplicates do not appear in search results, but they however are included in facet counts, which can be sometimes confusing for the users. This is a limitation of the index.
Example: The user narrows a query to one document that has a duplicate. Only one document appears in search results, but the facet count is 2.
The default value is false.

Markup configuration example(s) :
data-enable-duplicate-filtering='true'
data-enable-duplicate-filtering='false'

enableHistory

enableHistory: boolean
  • Defined in src/ui/SearchInterface/SearchInterface.ts :71

Specifies whether your search interface allows users to navigate in the search history using the browser back/forward buttons.
When enabled, the search interface saves the state of the current query in the hash portion of the URL.
For example #q=foobar.
The default value is false.

Markup configuration example(s) :

filterField

filterField: IFieldOption
  • Defined in src/ui/SearchInterface/SearchInterface.ts :109

Specifies the name of a field to use as a custom filter when executing the query (also referred to as 'folding').
Setting this option causes the index to return only one result having any particular value inside the filter field. Any other matching result is 'folded' inside the childResults member of each JSON query result.
This feature is typically used with threaded conversations to include only one top-level result per conversation. Thus, the field specified in this option typically is a value unique to each thread that is shared by all items (e.g.: posts, emails, etc.) in the thread.
This is obviously an advanced feature. Instead, look into using the Folding component, which covers a lot of different use cases.
By default none is added

Markup configuration example(s) :
data-filter-field='@foo'

maximumAge

maximumAge: number
  • Defined in src/ui/SearchInterface/SearchInterface.ts :171

Specifies the maximum age in milliseconds that cached query results can have in order to be used (instead of performing a new query on the index).
If cached results are available but are older than the specified age, a new query will be performed on the index.
On high-volume public web sites, having a larger maximum age can greatly improve query response time at the cost of result freshness.
By default, the Coveo Search API will determine the cache length. This typically takes 15 minutes.

Markup configuration example(s) :
data-maximum-age='10'

pipeline

pipeline: string
  • Defined in src/ui/SearchInterface/SearchInterface.ts :164

Specifies the name of the query pipeline to use for the queries. If not specified, the default value is default, which means the default query pipeline will be used.
You can use this parameter when your index is in a Coveo Cloud organization where you created pipelines (see https://onlinehelp.coveo.com/en/cloud/creating_and_managing_query_pipelines.htm).
Default value is 'default'.

Markup configuration example(s) :
data-pipeline='foo'

timezone

timezone: string
  • Defined in src/ui/SearchInterface/SearchInterface.ts :139

Specifies the timezone in which the search interface is loaded. This allows the index to recognize some special query syntax.
This must be an IANA zone info key (aka the Olson time zone database). For example : 'America/New_York'.
By default, we use a library that tries to detect the timezone automatically.

Markup configuration example(s) :
data-timezone='foo'
